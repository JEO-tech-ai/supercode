## 4. 핵심 차이점 상세

### 4.1 설계 철학

```
SuperCode: "명확하고 제어 가능한 시스템"
  - 작업 지향 (Task-oriented)
  - 예측 가능한 동작
  - 코드 기반 확장

Oh-My-OpenCode: "LLM 추론 기반 자동 오케스트레이션"
  - 메타데이터 기반
  - 동적 프롬프트 생성
  - 설정 기반 확장
```

### 4.2 에이전트 위임 흐름

**SuperCode**:
```
classifyRequest() → createPlan() → executePlan()
  ↓
규칙 기반 라우팅 → Task 객체 생성 → BackgroundManager 실행
```

**Oh-My-OpenCode**:
```
Sisyphus LLM → Phase 0-3 워크플로우
  ↓
프롬프트 내 Delegation Table → LLM이 에이전트 선택 → 실행
```

### 4.3 도구 수준

| SuperCode | Oh-My-OpenCode |
|-----------|----------------|
| bash, file, search, todo | LSP 11개 + AST-Grep |
| 저수준 OS 명령 | 고수준 코드 분석 |
| 범용성 | IDE 수준 정밀도 |

## 6. 권장 사항

### SuperCode에서 OpenCode 패턴 도입 시

1. **프롬프트 기반 에이전트 선택** 고려
   - 현재 정규표현식 → LLM 추론으로 전환 가능
   - Delegation Table 메타데이터 시스템 도입

2. **토큰 갱신 재시도 로직** 추가
   - exponential backoff 적용
   - 에러 분류 세분화

3. **LSP 도구 통합** 고려
   - 현재 저수준 도구 → 고수준 코드 분석 추가

### OpenCode에서 SuperCode 패턴 도입 시

1. **다중 프로바이더 지원** 확장
   - AuthHub 패턴 도입
   - Claude, OpenAI 직접 지원

2. **웹 UI** 추가
   - Solid.js 기반 콘솔
   - 실시간 대시보드

3. **토큰 암호화** 강화
   - AES-256-GCM 적용
